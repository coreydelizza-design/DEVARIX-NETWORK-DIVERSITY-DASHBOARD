## Section 1 — Current-state inventory

### What this repo actually is

A **demo/showcase build**, not a working tool. It is a ~1,000-line React 18 + Vite single-page app (`dev-source/src/`), compiled by `vite-plugin-singlefile` into one self-contained `index.html` (~180 KB) and served statically via `serve` on Railway (`Dockerfile`, `package.json`). Six views behind a sidebar router in `App.jsx`:

| View | File | What it does today | Real or mock? |
|---|---|---|---|
| Portfolio | `views/Dashboard.jsx` | 180-site dashboard: grade distribution, worst-first list, region/grade/tier filters, remediation program rollup | Fully synthetic — sites generated by seeded RNG in `lib/syntheticData.js` |
| Site scorer | `views/Scorer.jsx` | Interactive scorer: 6 weighted domains, 15 criteria, live composite + risk flags | Real mechanic, but selections are session-only (React state, lost on reload) |
| Site drill-down | `views/SiteDetail.jsx` | Circuit pair record (carrier, loop, wire center, PoP, AS path), SVG path-topology diagram, "design assumption ledger" | Fully hardcoded demo content (`assumptions` array, `SiteDetail.jsx:3-10`) |
| Intake & validation | `views/Intake.jsx` | Site card, 3-circuit inventory with validation status, entrance vault cards, working circuit-ID parser | Circuits hardcoded (`Intake.jsx:5-9`); only the parser is a live mechanic |
| Outage TCO | `views/Tco.jsx` | Interactive outage-exposure vs. remediation-cost calculator: payback, 3-yr net benefit | Real, working, session-only |
| Reports | `views/Reports.jsx` | Report catalog (4 report types described) + one rendered executive-scorecard preview | Preview computed from synthetic data; export explicitly stubbed ("export to PDF in a full build", `Reports.jsx:58`) |

### Data model

There is **no data model in the persistence sense**. There is no `localStorage`, `sessionStorage`, or IndexedDB usage anywhere in the repo (verified by search). All state is:

- **Generated:** `lib/syntheticData.js` builds 180 sites from a seeded RNG — `{id, city, region, tier, score, grade, flags[]}`. Flags are `[text, domain, impact]` tuples drawn from a 12-entry `flagPool`.
- **Hardcoded:** circuit inventories (`Intake.jsx:5-9`), circuit pair details and AS paths (`SiteDetail.jsx` kv tables), the assumption ledger (`SiteDetail.jsx:3-10`), entrance vault cards.
- **Ephemeral:** scorer selections and TCO slider values live in `useState` and vanish on reload.

There is no concept of engagement, client, circuit pair as a first-class entity, layer, evidence record, or finding. "Site" is the only entity, and it is synthetic.

### Scoring logic

`lib/scoringModel.js` — a weighted-domain attestation scorer:

- 6 domains: Local loop 25%, Backbone/transport 20%, Data center facility 15%, Edge routers 15%, Cloud connectivity 15%, Serving wire center 10%.
- Each criterion option = `[label, points 0–2, riskFlag|null]`. Domain score = points/max × 100; composite = weighted sum (`computeScore`, `scoringModel.js:122-139`).
- Grade bands (`gradeOf`): ≥85 Resilient, ≥65 Partially diverse, ≥40 Exposed, <40 Critical.
- Notably, many criteria already encode a **three-state epistemic ladder**: *same/shared* (0 pts) → *unknown/unvalidated* (1 pt) → *validated diverse* (2 pts). E.g. "Different carriers, unvalidated" vs. "validated diverse"; "Wire center: Same / Unknown / Different". The DNA of evidence-awareness exists — but it is collapsed into a number and carries no evidence reference.

### Intake flow

Display-only. The Intake view shows one hardcoded site (DEN-014) and three hardcoded circuits with status pills (Validated / Carrier mismatch / Unverifiable). The one genuine mechanic is `lib/circuitParser.js`: a Telcordia CLCI serialized-format parser with a 10-entry hardcoded CLLI table and 7 facility codes. It extracts serial, facility, A/Z CLLIs. There is **no form to enter or save a circuit**.

### Report/export capability

None. The Reports view is a catalog of four *described* report types plus one rendered HTML preview computed from synthetic portfolio stats. The export pill says it plainly: "Generated document · export to PDF in a full build." No print stylesheet, no PDF generation, no CSV/JSON export.

### Branding

- **In-app brand is already neutral:** "PATHSTEAD / Network diversity assurance" (`App.jsx:31-32`), page title "Pathstead — Network Diversity Assurance" (`dev-source/index.html`).
- **GTT appeared in exactly three places, all repo-level:** the repository name, the top-level `README.md` title, and the top-level `package.json` name. Zero GTT references in application source. *(Remediated after this review was written: the README title and package name were renamed to Pathstead in this same PR; the repository name itself can only be changed in GitHub Settings.)*
- Carrier names in demo content are genericized ("Carrier one/two"), ASNs use private ranges (64512+).

### Storage approach

None (see Data model). **This corrects a premise in the strategic context:** "storage remains localStorage" implies a localStorage layer exists; it does not. The localStorage persistence layer itself is net-new P0 work.

### Other observations relevant to the review

- The **design assumption ledger** (`SiteDetail.jsx`) is the closest existing analog to evidence grading: each hardcoded assumption carries a status (validated/violated/unvalidated), a note, and an evidence string with a date ("KMZ pair · Mar 2026", "LOA/CFA · Mar 2026"). The *concept* the business needs is already present as demo copy.
- Report preview copy already speaks the degradation language: "Evidence older than 12 months degrades to unvalidated at the next review cycle; 41 assumptions are scheduled to expire before Q4" (`Reports.jsx:55`). Again: concept present, mechanism absent.
- The build pipeline (edit `dev-source` → `npm run build` → copy `index.html` to root) is manual and drift-prone, though currently in sync.

## Section 2 — Alignment gap analysis

### 2.1 Evidence grading model — **CONFLICTS**

**What exists.** Two partial analogs:

1. The scoring model's option ladders (`scoringModel.js`) encode shared / unknown / validated states per criterion — semantically close to SHARED_FATE_CONFIRMED / (CLAIMED_UNVERIFIED or UNKNOWN) / VERIFIED_DIVERSE — but immediately collapse them into points. No evidence_ref, no confidence_note, no per-pair-per-layer structure (the scorer is per *site*, and the "pair" is implicit).
2. The assumption ledger (`SiteDetail.jsx:3-10`) has per-claim status + evidence + note — structurally the right shape — but it is hardcoded demo content with three states, not four: it does not distinguish CLAIMED_UNVERIFIED (carrier asserted, no record) from UNKNOWN (no data obtainable from anyone). That distinction is a first-class product requirement because UNKNOWN is itself a billable finding.

**The conflict.** The composite score is the headline output of the tool (Dashboard metric, Scorer metric, report preview), and it is exactly what the strategic context warns against: a numeric attestation without evidence references. Left as-is, it undermines the defensibility claim of the audit deliverable.

**Proposed coexistence (per constraint — neither model discarded).** Invert the derivation: grades become the primary data; the score becomes a *view* over grades.

- Per circuit pair, per layer: `{grade: VERIFIED_DIVERSE|CLAIMED_UNVERIFIED|UNKNOWN|SHARED_FATE_CONFIRMED, evidence_ref, confidence_note, verified_date}`.
- Map grades to the existing 0–2 point scale: VERIFIED_DIVERSE=2, CLAIMED_UNVERIFIED=1, UNKNOWN=0.5 (or 1 — a calibration decision; UNKNOWN should arguably score *below* a carrier claim), SHARED_FATE_CONFIRMED=0. The existing weighted-domain math (`computeScore`) then runs unchanged on derived points, preserving the executive-friendly composite and the grade bands, while every point traces to an evidence record.
- The existing option ladders become the *layer taxonomy seed*: each scoring criterion maps to one of the seven layers of requirement 2.
- Report as: score for executives, grade matrix for the defensible findings body.

**Verdict: CONFLICTS** (repairable by derivation, not replacement).

**Verdict: MISSING** (a functional intake module must be built; the current view is a visual stub with one reusable mechanic).

### 2.3 Cross-examination engine — **MISSING**

None of the seven checks exists as executable logic. What exists is *narrative proof the team understands the checks* — hardcoded demo content that describes their outcomes:

- Check 1 (wire center CLLI match): SiteDetail shows two different wire centers; the scoring model has a same/unknown/different criterion. No comparison code.
- Check 2 (access vendor match on Type II): Intake demo note — "Billed as carrier two but ID format and CLLI indicate underlying carrier one facility — diversity claim suspect" — is exactly this check, performed by a human and pasted in as copy.
- Check 3 (NNI match): scoring criterion label only.
- Check 4 (KMZ route overlap): assumption-ledger copy references KMZ evidence; nothing tracks receipt or result.
- Check 5 (building entrance match): Intake's vault cards narrate a shared-entrance finding; hardcoded.
- Check 6 (POP/node/card match): SiteDetail's whole topology diagram dramatizes this finding (same building, floors 6 and 9); hardcoded.
- Check 7 (same-ASN egress): SiteDetail AS-path rows would be the input; hardcoded.

Checks 1, 2, 3, 6, 7 are pure string comparisons on structured fields — trivially automatable client-side **once requirement 2's schema exists**. The engine is blocked entirely on intake, not on any technical difficulty. Checks 4 and 5 need result-tracking fields only (external GIS workflow and site walk respectively), which is a schema concern, not an engine concern.

**Verdict: MISSING** (fully feasible client-side; dependency is the intake schema).

### 2.4 Engagement workflow layer — **MISSING (net-new surface)**

Nothing tracks the engagement. No LOA status, no per-carrier records requests, no request/response/follow-up cycle, no NDA tracking, no timeline. The words LOA and CFA appear twice, both as demo copy (`circuitParser.js:49` — "manual validation queue with an LOA/CFA request to the billed carrier"; `SiteDetail.jsx:7` — "LOA/CFA · Mar 2026" as an evidence string).

**Can any current surface host it?** No. All six views are site/portfolio-scoped; the engagement is a client/carrier-scoped concern with a time dimension. It needs a seventh view (or an "Engagement" home that wraps the others). The sidebar router in `App.jsx` makes adding a view trivial (add to `nav` array + one conditional), and the existing card/pill/row-list components (`components/ui.jsx`, `styles.css`) are sufficient UI vocabulary for a tracker (request rows with status pills mirror the Intake circuit rows exactly).

The 8-item ask list (DLR, serving wire center, access provider, NNI, KMZ, POP/node, entrance facility, written diversity attestation) should be a constant in the data layer — it doubles as the checklist that drives per-carrier request completeness *and* maps 1:1 onto the evidence_source fields of requirement 2, which is the integration point: a received record satisfies a records-request item AND becomes an evidence_ref available to grading.

**Verdict: MISSING** — net-new surface, but low-friction to add within the existing architecture.

### 2.5 Report generation — **PARTIAL**

What exists: a Reports view whose *catalog copy* describes almost exactly the right deliverables — "Site validation report … the artifact for auditors, insurers, and regulators" and "Evidence audit / exceptions — everything unvalidated, expired, or mismatched" are the audit deliverable and the UNKNOWN/expiry queue by other names. The rendered scorecard preview proves the pattern (reports as generated views over the data model, not authored documents — the right architecture, stated in the view's own subtitle). The TCO calculator (`Tco.jsx`) already computes the cost-of-inertia framing (exposure, avoided loss, payback, 3-yr net) — it just isn't embedded in any report.

What's missing: everything evidentiary. No per-site/per-pair/per-layer grade tables (blocked on requirements 1–2), no VERIFIED/CLAIMED/UNKNOWN/SHARED_FATE counts (blocked on requirement 1), no export path at all — the pill admits PDF export is stubbed. Client-side print-to-PDF via a print stylesheet is fully achievable without a backend and is the pragmatic P0 answer.

**Verdict: PARTIAL** — right report architecture and cost framing exist; the evidence-graded substance and the export mechanism do not.

### 2.6 Registry data model — **MISSING**

There is no data model to normalize (see Section 1). Assessed as "what must the new model include so the future DB refactor is a lift, not a redesign":

- **Canonical entity tables from day one:** CLLIs (seeded by `clliTable`, which is the only registry-shaped artifact in the repo — 10 entries, hardcoded), ASNs, access vendors, NNI IDs, route segments, POPs. Circuit records should hold *references* (IDs) into these tables, never free-text strings — free-text carrier/CLLI entry is the single biggest threat to cross-engagement accumulation.
- **Engagement-scoped facts, registry-scoped entities:** grades and evidence belong to an engagement; wire centers, NNIs, and shared-fate confirmations are engagement-independent knowledge. The model should separate the two namespaces now, so "export registry layer" is a filter, not a migration.
- **Shared-fate assertions as first-class records:** `{entity_a, entity_b, layer, relation: shared_fate|verified_diverse, evidence_ref, engagement_id, verified_date}` — this is the compounding asset; it must not live only inside per-engagement findings.
- **Normalization discipline:** uppercase/trim CLLIs (the parser already uppercases), canonical vendor naming, stable internal IDs.

localStorage remains viable at engagement scale (dozens of sites × a few circuits × 7 layers is well under quota); JSON export/import per engagement covers backup and the cross-engagement carry-forward until the milestone-gated database.

**Verdict: MISSING** (nothing exists, but nothing conflicts — greenfield with one seed asset, `clliTable`).

### 2.7 Degradation / expiry — **MISSING**

No temporal dimension exists in any data structure. The *concept* appears twice as demo copy: evidence dates in the assumption ledger ("Site survey · Jun 2026") and the report preview's degradation sentence ("Evidence older than 12 months degrades to unvalidated…; 41 assumptions are scheduled to expire before Q4"). The synthetic data model has no date field of any kind; the scorer has no notion of when a selection was made.

Required and absent: `verified_date` on every grade record, a per-layer or per-evidence-type re-verification interval (a re-groom invalidates a wire-center fact faster than a building entrance fact), a derived `stale` state that degrades VERIFIED_DIVERSE → CLAIMED_UNVERIFIED (or flags it), and an expiry queue view — which the Reports catalog already describes as the "Evidence audit / exceptions" report. This requirement is also the technical seed of the monitoring subscription: the expiry queue *is* the recurring-revenue product surface.

**Verdict: MISSING** (concept present in copy; no mechanism).

### 2.8 Branding neutrality — **PARTIAL** (better than the prompt assumed)

Complete inventory of GTT-specific references as found at review time (`943bfcc`):

| # | Location | Reference | Decouple effort |
|---|---|---|---|
| 1 | Repository name | `GTT-NETWORK-DIVERSITY-DASHBOARD` | Rename repo (out of code's control; cosmetic) |
| 2 | `README.md:1` | "# GTT Network Diversity Dashboard" | One line |
| 3 | `package.json:2` (root) | `"name": "gtt-network-diversity-dashboard"` | One line |

That is the entire GTT footprint. The application itself ships a different brand: "PATHSTEAD" hardcoded in `App.jsx:31`, "Pathstead — Network Diversity Assurance" in `dev-source/index.html` `<title>` and `dev-source/README.md:1`. No GTT theming, colors, logos, or assumptions exist in source; carrier names are genericized.

**Effort to decouple into a config layer: Small.** A single `brandConfig` module (name, subtitle, page title, optionally palette tokens — `styles.css` already uses CSS variables throughout) consumed by `App.jsx` and injected into the HTML title covers 100% of in-app branding. The strategic fork (inside-GTT vs. independent) then reduces to one config file plus a repo rename. Per constraints, nothing was renamed.

**Verdict: PARTIAL** — in-app branding is already neutral-but-hardcoded (Pathstead); no config layer exists; GTT references were confined to three repo-level strings.

## Section 4 — Proposed build-prompt sequence

Discrete, one-feature-per-prompt Claude Code prompts, in dependency order. All target `dev-source/src/` with a rebuild of `index.html` as the final step of each. P0 = prompts 1–9; P1 = prompts 10–12.

1. **Data layer & persistence.** Create `lib/schema.js` + `lib/store.js`: versioned localStorage store with entities engagement/client/site/circuit/pair/layer-grade and canonical registry tables (CLLI, ASN, access vendor, NNI, route segment, POP); include JSON export/import of a full engagement.
   *Acceptance: an engagement created in the UI survives a page reload, and exports/imports losslessly as JSON.*

2. **Engagement & site CRUD.** Add an Engagement view (nav item 0): create/select engagement, client record, sites with normalized addresses; all other views become engagement-scoped.
   *Acceptance: two engagements can coexist with separate sites and switching between them swaps all view data.*

3. **Per-circuit intake, part 1 — identity + local loop.** Rewrite Intake as a persisted per-circuit form: identity fields (circuit_id, carrier, service_type, site, address) and local-loop fields (on-net/Type II, access provider, access circuit ID, wire center CLLI), each layer with evidence_source; wire the existing parser to auto-fill facility and A/Z CLLIs.
   *Acceptance: pasting a CLCI-format ID auto-populates fields, and a saved circuit reloads with every field intact.*

4. **Per-circuit intake, part 2 — remaining layers.** Add building entrance (conduit/demarc/riser/power), NNI (location CLLI, NNI ID; shown for Type II only), route (KMZ received y/n, file ref, overlap segments), POP/node (POP CLLI, router/node, shelf/card/port), and logical (egress ASN, BGP multihoming, traceroute divergence) — each with evidence_source.
   *Acceptance: every field in the requirement-2 schema exists on the form, persists, and NNI fields appear only when Type II is selected.*

5. **Circuit pairs & four-grade evidence model.** Add pair definition per site (choose 2 circuits) and a per-layer grade matrix: VERIFIED_DIVERSE / CLAIMED_UNVERIFIED / UNKNOWN / SHARED_FATE_CONFIRMED, each grade requiring evidence_ref + confidence_note and stamping verified_date.
   *Acceptance: a pair displays a 7-layer grade matrix and a grade cannot be saved without an evidence_ref.*

6. **Derived scoring.** Refactor `scoringModel.js` so the weighted composite is computed FROM the pair's layer grades (VERIFIED=2, CLAIMED=1, UNKNOWN=0.5, SHARED_FATE=0 on the existing 0–2 criterion scale), keeping domains/weights/grade bands; Scorer view becomes a read-out with an evidence-coverage % alongside the score.
   *Acceptance: changing a single layer grade updates the composite, and the score panel shows what % of layers carry evidence refs.*

7. **Cross-examination engine.** Create `lib/crossExam.js` running checks 1 (wire center CLLI), 2 (access vendor on Type II), 3 (NNI ID), 6 (POP/node/card), 7 (egress ASN) as automated comparisons on every pair, proposing SHARED_FATE_CONFIRMED where fields match; add tracked manual results for checks 4 (KMZ overlap) and 5 (entrance match).
   *Acceptance: entering the same serving wire center CLLI on both circuits of a pair auto-raises a shared-fate proposal on the wire-center layer.*

8. **Records-request tracker.** In the Engagement view: LOA status, NDA status, per-carrier request tracker seeded with the 8-item ask list (DLR, serving wire center, access provider, NNI, KMZ, POP/node, entrance facility, diversity attestation), request/response/follow-up states with dates, and a 60–90-day timeline strip; received items become selectable evidence_refs in grading.
   *Acceptance: marking "serving wire center" received for a carrier makes it available as an evidence_ref when grading that carrier's circuits.*

## Appendix — OUT OF SCOPE, OBSERVATIONS ONLY

Not requirements; noted in passing and deliberately excluded from the plan above.

- **Build-pipeline drift risk.** The manual `dev-source` → build → copy-`index.html`-to-root flow will eventually ship a stale bundle. A one-line build script that copies the artifact up (or a CI step) would remove the risk. Currently in sync.
- **Site-walk capture.** Check 5 evidence originates on-site; a phone-usable entrance-facility capture form (photos referenced by filename) would strengthen chain of evidence. localStorage size makes photo *storage* impractical — references only.
- **Traceroute ingestion.** Check 7's inputs (egress ASN, traceroute divergence) could be semi-automated by pasting raw traceroute/BGP output for parsing, same pattern as the circuit-ID parser.
- **Engagement templates.** After 2–3 audits, the 8-item ask list will likely grow per-carrier variants (each carrier's records team responds differently); a per-carrier template concept would fit the registry layer naturally.
- **The TCO calculator is underused strategically.** It is currently a standalone toy, but its output is the renewal argument for the monitoring subscription ("here is what expiring evidence re-exposes you to"). Worth revisiting in P2.
- **`serve` dependency in root `package.json`** duplicates what the Dockerfile installs globally; harmless, but one of them is redundant.
